# VibePilot Project Rules

You are working on VibePilot -- a sovereign AI execution engine. Read these rules before every task. They exist because agents keep making the same mistakes. Each rule cost real time to learn.

## Absolute Rules

1. **NEVER hardcode anything.** Models, connectors, routing, agents -- all in config JSON files under `config/`.

2. **NEVER hunt for .env files.** Credentials live in Supabase vault. Period. No .env files exist. Use vault.

3. **NEVER guess -- check first.** Query knowledge.db, read existing code, check what's there before inventing anything.

4. **Schema migrations have ONE path. Use it. No exceptions.**
   Supabase REST/curl/RPC/psql ALL fail for DDL. The ONLY working path:
   - `ls ~/VibePilot/docs/supabase-schema/` -- find the NEXT unused number
   - Write `NNN_name.sql` with DROP IF EXISTS for idempotency
   - Commit and push to GitHub main
   - `cd ~/vibepilot && git pull`
   - Give human the direct GitHub link + "paste into SQL Editor and Run"
   - Do NOT try any other method. Do NOT reuse existing numbers.

5. **ALWAYS push to GitHub.** Local-only work gets lost.
   Local config files MUST be backed up to `backup/` in the repo.
   If it only exists on this machine, it doesn't exist.

6. **NEVER take shortcuts or "do it later."** If you can't do it properly now, stop and say so.

7. **Go SERVES VibePilot's design. It does NOT invent new processes.**
   Go is plumbing. Supabase is the contract. Dashboard shows what state IS.
   Go conforms to both. Not the other way around.

8. **Governor is a systemd user service.** `systemctl --user` and `journalctl --user`. Not sudo.

9. **Roles are defined in config, not guessed.**
   - Supervisor: reviews plans, reviews output, approves/escalates
   - Council: reviews complex plans and architecture
   - Human (Vibes): visual UI/UX decisions, strategic architecture yes/no. NOT technical. Does NOT code/merge/debug.
   - Maintenance: ONLY agent with git write access
   - Merge system: fully automated, zero human involvement

10. **Read before you code.** Query knowledge.db first. 5 minutes of reading saves hours of rewriting.

## The Three Knowledge Systems (no overlap)

1. **lean-ctx / map.md** -- Go function signatures only. Compressed view of all Go code. For "what functions exist in the router package."
2. **jCodeMunch / index.db** -- ALL code structure: Go + SQL + YAML + JSON + Python + shell. 1974 symbols. For "find where claim_task is defined" across any file type.
3. **knowledge.db** -- Prose and structure: rules, prompts, configs, docs, SQL schema definitions, pipeline stages. For "what are the tier0 rules" or "show me the claim_task function body" or "what does the code pipeline look like."

## Task Lifecycle & Git Architecture

**READ VIBEPILOT_WHAT_YOU_NEED_TO_KNOW.md Section 6 for the canonical pipeline.**

The pipeline has 11 stages. The critical distinctions:
- **Supervisor** reviews: plan quality (step 3) + output alignment (step 6)
- **Council** reviews: complex plan architecture (step 3 only, supervisor triggers)
- **Testing handler** validates: code correctness (step 7) -- three layers: output validation, semgrep, native tests
- **Module branches**: task branch → module branch (after individual test) → project main (after module integration test)
- **Testing is NEVER skipped.** It is fixed to be project-aware, not skipped.

Current state of task lifecycle implementation is in `docs/ARCHITECTURE_TASK_LIFECYCLE.md`.
Worktree code exists in `governor/internal/gitree/` (NOT YET fully wired).

- `~/VibePilot/` -- DEVELOPMENT copy (primary working directory)
- `~/vibepilot/` -- RUNNING copy (compiled binary + systemd service)
- Both must stay synced on main branch

## Architectural Thinking (MANDATORY)

Before ANY code change, load the `vibepilot-thinking` skill. It encodes:
- Foundation-first reasoning (upstream dependencies are not "nice-to-have")
- Coherent unit boundaries (never commit half of an interdependent change)
- Whackamole anti-pattern (fix root causes, not symptoms)
- Consider-before-build (15 min thinking < 2 hours fixing)

If you catch yourself saying "let's just commit what we have" or "we can do this later" -- STOP, load the skill, recalibrate.

## Post-Task Discipline (NON-OPTIONAL)

After completing any task, update what changed:
1. CURRENT_STATE.md -- sections affected by your work
2. WYNTK -- if architecture, paths, or procedures changed
3. tier0-static.md -- if you discovered a new rule agents keep violating
4. TODO.md -- mark done, add new items
5. Commit and push -- local-only work gets lost
6. The .context/ layer auto-rebuilds on commit. But source docs only update if you update them.

## Human Boundaries

- Human is NOT technical. Never assume technical knowledge.
- Human sees visual outputs and makes aesthetic/UX decisions.
- Human makes strategic architecture decisions after Council review.
- Human does NOT code, merge, debug, review code, or fix merge conflicts.
- Respect human time: batch questions together, don't ask one at a time.
