# The .context/ Knowledge Layer

**Purpose:** Persistent, compressed, queryable memory that survives agent swaps. Every agent that works on VibePilot starts here. This doc explains what it is, why it exists, what it replaced, and how to maintain it.

---

## The Problem It Solves

VibePilot is worked on by multiple agents (Hermes, Claude Code, Codex, OpenCode, Kimi, etc). Agents get swapped frequently. Each new agent starts with zero context about the project. Without a persistent knowledge layer:

- Agents repeat the same mistakes (hardcoding, hunting for .env files, wrong systemctl commands)
- Each session wastes 30+ minutes rediscovering project structure
- Decisions and patterns from previous sessions are invisible
- The 14k-line Go codebase can't fit in a single model's context window

The .context/ layer is how we solve this. It's a compressed, auto-generated, queryable knowledge base that any agent can read in seconds.

## Tool Evaluation: Why We Built Our Own

We evaluated three tools before deciding to build a custom integrated system:

### What Each Tool Does

| Capability | lean-ctx 3.1.2 | jCodeMunch 1.43 | jDocMunch 1.8 | Hermes (agent built-in) |
|---|---|---|---|---|
| **Code compression** | 90+ compression patterns, shell hooks, read modes (map/signatures/aggressive/entropy) | Code indexing via AST/symbols, SQLite index.db | Document section extraction, tarball output | Memory system, session recall, skill persistence |
| **Code indexing** | No index DB -- compresses on-the-fly | Yes -- symbols, imports, call graph into SQLite | No code indexing | No code indexing |
| **Doc extraction** | No -- shell command output only | No docs | Yes -- markdown headings/sections to JSON/tarball | No doc extraction |
| **Queryability** | CLI commands only, no persistent DB | SQLite index.db (code symbols) | Tarball output (not queryable) | Memory/key-value, session search |
| **Rules/principles** | No | No | No | Memory entries (unstructured) |
| **Config parsing** | No | No | No | No |
| **Auto-rebuild** | Manual CLI usage | Manual or watch mode | Manual | Manual |
| **Cost** | Free (open source) | Free (open source) | Free (open source) | Built-in |
| **Output format** | Compressed text to terminal | SQLite DB | JSON/tarball | In-memory |

### What Was Missing

No single tool covered what we needed:

1. **lean-ctx** is great at compressing shell output and file reads, but it doesn't build a persistent index or extract docs/rules/configs
2. **jCodeMunch** builds the best code index (symbols, imports, call graph) but ignores docs, rules, configs, and prompts entirely
3. **jDocMunch** extracts document sections but outputs to a clunky tarball that isn't queryable, and doesn't touch code or configs
4. **Hermes memory** survives sessions but is unstructured text with a tight character limit -- not queryable, not compressed

### Our Solution: Combined Pipeline

We use all three tools together, plus our own custom Python script that replaces jDocMunch with something better:

```
build.sh orchestrates:
  1. lean-ctx read -m map  →  map.md     (compressed function signatures for every Go file)
  2. jCodeMunch index      →  index.db   (SQLite: symbols, imports, call graph, AST)
  3. build-knowledge-db.py →  knowledge.db (SQLite: rules, prompts, configs, docs sections)
  4. Auto-assembly         →  boot.md    (Tier 0 rules first, then structure, then status)
  5. Auto-assembly         →  meta.yaml  (what was generated, versions, timestamps)
```

**Why build-knowledge-db.py replaces jDocMunch:**
- jDocMunch outputs to a tarball. You'd have to unpack it to search it. Useless for agents.
- Our script outputs to SQLite. Agents query it directly with standard SQL.
- Our script parses tier0-static.md as the primary rules source (hand-crafted, curated) -- jDocMunch has no concept of rules.
- Our script extracts prompts, configs, and docs into separate queryable tables.
- 4 tables: `rules` (priority-ranked), `prompts` (role-tagged), `configs` (key fields), `docs` (full-text searchable sections).

## What's In .context/

```
.context/
├── boot.md              # Agent orientation (~2,800 tokens). START HERE.
│                        # Tier 0 rules first, then codebase tree, then status.
│                        # Auto-generated. DO NOT edit directly.
│
├── map.md               # Compressed code map (~12,000 tokens).
│                        # Every function signature, type, import across all Go files.
│                        # Generated by lean-ctx read -m map.
│
├── index.db             # jCodeMunch SQLite (3.5MB).
│                        # Code symbols, imports, call graph, AST data.
│                        # sqlite3 .context/index.db ".tables"
│
├── knowledge.db         # Our custom SQLite (2.5MB). THE KEY FILE.
│                        # 4 tables: rules, prompts, configs, docs.
│                        # Generated by build-knowledge-db.py.
│
├── meta.yaml            # Generation metadata (commit, branch, tool versions).
│
├── build.sh             # Orchestrator. Runs all tools, assembles output.
│
├── sync.sh              # Cron job script. Rebuilds + auto-pushes to GitHub.
│
├── tools/
│   ├── tier0-static.md  # SINGLE SOURCE OF TRUTH for rules/principles/roles.
│   │                    # Hand-crafted. build-knowledge-db.py reads this first.
│   │                    # EDIT THIS FILE to change rules. Do NOT edit boot.md.
│   │
│   ├── build-knowledge-db.py  # Our custom Python. Replaces jDocMunch.
│   │                          # Parses tier0 + prompts + configs + docs into SQLite.
│   │
│   ├── tier0.py         # Helper for tier0 parsing.
│   ├── install.sh       # Installs lean-ctx + jCodeMunch via pipx.
│   └── json-desc.sh     # Extracts descriptions from JSON config files.
│
├── DECISION_LOG.md      # Historical decision records.
├── guardrails.md        # Agent safety guardrails.
├── ops_handbook.md      # Operational procedures.
├── quick_reference.md   # Quick command reference.
└── agent_protocol.md    # Agent communication protocol.
```

## How To Use .context/

### For any agent starting a new session:

```
1. Read boot.md first (~2,800 tokens). Gets you oriented.
2. For specific questions, query knowledge.db:
   sqlite3 .context/knowledge.db "SELECT title,content FROM rules WHERE priority='critical'"
   sqlite3 .context/knowledge.db "SELECT name,role FROM prompts WHERE name LIKE '%supervisor%'"
   sqlite3 .context/knowledge.db "SELECT title,file_path FROM docs WHERE title LIKE '%vault%'"
3. For code details, check map.md or index.db.
```

### When you make changes:

```
1. Edit tier0-static.md if you're adding/changing rules or principles.
2. Everything else (boot.md, map.md, knowledge.db) auto-rebuilds:
   - On every git commit (pre-commit hook)
   - Daily at 2am (cron job)
   - Manually: bash .context/build.sh
```

## How It's Automated

| Trigger | What Happens | How |
|---|---|---|
| `git commit` | Pre-commit hook detects Go/config changes, rebuilds .context/, auto-stages | `.git/hooks/pre-commit` |
| Daily 2am | sync.sh rebuilds + commits + pushes to GitHub | `crontab -l` |
| Manual | `bash .context/build.sh` from repo root | Anytime |
| Fresh machine | `bash .context/tools/install.sh` then `bash .context/build.sh` | Recovery |

---

## Architecture Decision: Three Tiers of Knowledge

| Tier | What | Source | Size | Purpose |
|---|---|---|---|---|
| **Tier 0** | Non-negotiable rules | tier0-static.md (hand-crafted) | ~4,500 bytes | Prevent repeated agent mistakes |
| **Tier 1** | Rules, prompts, configs | knowledge.db (auto-built) | 2.5MB | Queryable project knowledge |
| **Tier 2** | Full documentation sections | knowledge.db (docs table) | ~3,000 sections | Deep reference when needed |

Agents load Tier 0 first (in boot.md), then query Tier 1/2 as needed. They never need to load everything at once.

---

## Recovery

If .context/ gets corrupted or you're on a fresh machine:

```bash
git clone https://github.com/VibesTribe/VibePilot.git
cd VibePilot
bash .context/tools/install.sh    # installs lean-ctx + jCodeMunch via pipx
bash .context/build.sh            # rebuilds everything from repo sources
```

Everything in .context/ is auto-generated from repo contents. Nothing is lost if you have the repo.
